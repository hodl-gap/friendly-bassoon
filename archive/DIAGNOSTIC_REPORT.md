# Diagnostic Report: Belief-Space Architecture Evaluation

**Date**: 2026-02-06
**Evaluator**: Developer Agent (Diagnostic Mode)
**Scope**: End-to-end pipeline (Database Manager → Retriever → BTC Intelligence)

---

## GOAL Achievability Verdict

### **PARTIALLY ACHIEVABLE**

The system **can collect and preserve** multiple contradictory logic chains through retrieval, but **collapses them to a single direction** at the final output layer. The belief-space representation exists transiently during LLM analysis but is discarded before output.

---

## Blocking Limitations (Ranked)

### 1. **CRITICAL: Single-Direction Output Enforcement** (BTC Intelligence)

**Location**: `impact_analysis.py` lines 94-100, `states.py` line 33

The final state stores only:
```python
direction: str  # Single value: BULLISH / BEARISH / NEUTRAL
```

The LLM is instructed to generate scenarios with different directions, but the parser extracts only `PRIMARY_DIRECTION`. All alternative scenarios are **discarded after LLM generation**.

**Classification**: Foundational for output layer, but **incremental to fix** (add `scenarios` field to state, modify parser)

---

### 2. **CRITICAL: Scenario Details Not Parsed or Stored**

**Location**: `impact_analysis.py` lines 105-169 (parser)

The parser uses regex to extract only:
- `PRIMARY_DIRECTION`
- Single `CONFIDENCE` block
- Single `RATIONALE`

The `SCENARIOS:` section generated by the LLM exists in the response text but is **never parsed into structured data**. It vanishes after the LLM call.

**Classification**: Incremental (parser modification)

---

### 3. **HIGH: No Per-Scenario Confidence Model**

**Location**: `impact_analysis_prompts.py` lines 184-188

Confidence is a single object:
```python
{
    "score": 0.72,
    "chain_count": 3,
    "source_diversity": 2,
    "strongest_chain": "..."
}
```

There is no structure for:
- Confidence per scenario
- Chain count per belief path
- Source diversity per interpretation

**Classification**: Incremental (extend confidence schema)

---

### 4. **MEDIUM: Polarity Implicit in Logic Chains**

**Location**: Database Manager `data_opinion_prompts.py`

Logic chain steps store:
```json
{
    "cause": "TGA increase",
    "effect": "bank reserves drain",
    "mechanism": "..."
}
```

Polarity (up/down) is **implicit in natural language**, not an explicit field. The `liquidity_metrics` array has a `direction` field, but logic chains do not.

This means: The system can represent "TGA increase → BTC pressure" but cannot explicitly tag the chain as "BEARISH" without LLM interpretation.

**Classification**: Incremental (add `polarity` field to chain schema)

---

### 5. **MEDIUM: Contradiction Detection is Post-Hoc, Not Structural**

**Location**: Retriever `answer_generation.py` lines 867-896

Contradictions are detected via a separate LLM pass (Stage 3) and output as:
```
## CONTRADICTING EVIDENCE
**CONTRADICTION:** [text]
**IMPACT:** High/Medium/Low
```

But this is **narrative text**, not structured belief branches. Contradictions are surfaced as "weakening evidence" against a consensus, not as first-class alternative belief paths.

**Classification**: Incremental (restructure contradiction output as scenario objects)

---

### 6. **LOW: No Explicit "Belief Branch" Data Structure**

The system represents chains as flat lists:
```python
state["logic_chains"] = [chain1, chain2, chain3, ...]
```

There is no grouping by:
- Belief branch / interpretation
- Outcome polarity
- Trigger origin

Chains leading to opposite outcomes sit in the same undifferentiated list.

**Classification**: Incremental (add grouping/tagging to chain list)

---

## Missing Capabilities (Abstract)

| Capability | Current State | Required For Goal |
|------------|---------------|-------------------|
| **Multi-direction output** | Single `direction` field | Store/display multiple scenarios with directions |
| **Scenario persistence** | LLM generates then discards | Parse and store scenario objects in state |
| **Per-scenario confidence** | Single confidence block | Confidence metadata per belief path |
| **Explicit chain polarity** | Implicit in text | `polarity: "BEARISH"` field on chains |
| **Belief-branch grouping** | Flat chain list | Group chains by outcome/interpretation |
| **Contradiction-as-scenario** | Narrative "weakening evidence" | Contradictions as first-class alternative paths |

## Rubric Evaluation

### I. Logical Chain Formation: **PASS**

The system produces explicit, ordered chains with structure:
```
cause [normalized] → effect [normalized] : mechanism
```

Multi-step chains supported. Evidence quotes mandatory.

### II. Branching From Single Trigger: **PARTIAL PASS**

- **Retrieval layer**: Multiple chains from same trigger preserved
- **LLM analysis**: Scenarios with different directions generated
- **Output layer**: **FAILS** — only PRIMARY_DIRECTION survives

The architecture **can branch** but **forces convergence** at output.

### III. Contradiction Preservation: **PARTIAL PASS**

- **Retriever Stage 3**: Contradictions explicitly detected and surfaced
- **BTC Intelligence**: LLM instructed to present diverging scenarios
- **Final output**: Contradictions collapsed to likelihood percentages, winner selected

Contradictions are **acknowledged** but **resolved** rather than preserved as coexisting beliefs.

### IV. Outcome Polarity Support: **PARTIAL PASS**

- **Implicit**: Cause/effect text contains directional meaning
- **Explicit**: `liquidity_metrics` has `direction` field
- **Logic chains**: No explicit `polarity` field
- **Output**: Single direction, not per-chain polarity

### V. Event, Time, and Magnitude Anchoring: **PASS**

- Evidence quotes mandatory with verbatim text
- Temporal context stored (`valid_from`, `valid_until`, `policy_regime`)
- Historical event detection fetches actual prices/magnitudes
- Knowledge gap filling retrieves specific dates and values

### VI. Reasoning Architecture: **PASS**

- Multi-stage pipeline with inspectable intermediate states
- Logic chains visible at each stage
- Confidence metadata shows chain count, source diversity
- Historical analog comparison explicit

### VII. Epistemic Coverage: **PARTIAL PASS**

- Multiple sources retrieved and preserved
- Chain-of-retrievals expands coverage
- Contradiction detection adds alternative views
- **BUT**: Final output is singular, not a belief space

## What the Current System Already Does Well

### 1. **Logic Chain Extraction & Linking**
- Normalized variable names (`cause_normalized`, `effect_normalized`) enable cross-chunk chain connection
- Multi-hop reasoning: A→B from Source 1 + B→C from Source 2 = A→B→C
- Dangling effect detection triggers follow-up retrieval

### 2. **Contradiction Detection Infrastructure**
- Dedicated Stage 3 in Retriever specifically looks for contradicting evidence
- Impact assessment (High/Medium/Low) on each contradiction
- This infrastructure could be repurposed for scenario generation

### 3. **Evidence Grounding**
- Verbatim quote requirement prevents hallucination
- QA validation checks extraction accuracy
- Historical event detection fetches actual market data (not just narratives)

### 4. **Temporal Awareness**
- Regime tagging (`QE`, `QT`, `reserve_scarce`, etc.)
- Valid date ranges on chains
- Warning when query year ≠ data years
- Extrapolation notes when using proxy data

### 5. **Scenario Generation in LLM**
- The prompt already instructs LLM to generate multiple scenarios with:
  - Separate causal chains
  - Likelihood percentages
  - Per-scenario directions
- This output exists — it just needs to be parsed and preserved

### 6. **Confidence Metadata**
- Chain count, source diversity, strongest chain tracked
- Provides foundation for per-scenario confidence extension

## Conclusion

The system is **architecturally capable of belief-space representation** through the retrieval layer and even generates multi-scenario outputs during LLM analysis. However, the **final output layer enforces single-direction convergence** by:

1. Parsing only `PRIMARY_DIRECTION` from LLM response
2. Storing only a single `direction` field in state
3. Displaying only one direction in CLI/JSON output

**The fix is incremental, not foundational:**
- The belief branches exist — they're just discarded at parse time
- Extending the parser and state schema would preserve them
- No core reasoning paradigm change required

**Verdict**: The GOAL is achievable with targeted modifications to the output layer, without architectural overhaul of the retrieval or chain extraction systems.
